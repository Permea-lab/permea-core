# Contributor Levels

## 1. Purpose

This document defines Permea Core's initial contributor level system.

The goal is to make contribution paths legible while separating GitHub participation, contributor credit, reviewer trust, maintainer rights, acknowledgements, and paper authorship.

## 2. Contributor Level System

Permea uses contribution levels to describe observed contribution roles, not personal status or permanent rank.

Levels are:

- descriptive, not automatic rights
- based on contribution type and review history
- bounded by claim, data, and release risk
- separate from paper authorship
- separate from institutional or employment status

## 3. Community Participant Levels

### Observer

Reads the project, follows discussions, or uses public docs without active contribution.

### Issue Reporter

Reports bugs, unclear docs, reproducibility concerns, claim-boundary concerns, or dataset/provenance questions.

### Documentation Contributor

Improves documentation, examples, checklists, terminology, or policy clarity.

### Code Contributor

Contributes scripts, package code, tests, tooling, or workflow improvements.

### Benchmark Contributor

Contributes benchmark contracts, task definitions, split policies, metric reviews, baseline definitions, or benchmark templates.

### Reproducibility Contributor

Improves rerun instructions, environment capture, run manifests, reproducibility checks, or artifact traceability.

### Dataset / Provenance Contributor

Contributes dataset source documentation, provenance manifests, licensing notes, source-to-row mapping, or public-safe release reviews.

### Claim Reviewer

Reviews README, manuscript, release, website, or benchmark language against the claim registry and evidence ladder.

### Scientific Reviewer

Reviews scientific framing, method interpretation, literature context, benchmark scope, limitations, and evidence-tier discipline.

### Maintainer

Has project stewardship responsibilities and may approve or merge scoped changes according to project governance.

Maintainer status is granted explicitly. It is not automatic.

## 4. Contribution Types and Expected Evidence

Expected contribution evidence may include:

- merged pull requests
- reviewed issues
- documented review comments
- reproducibility reports
- source-to-claim review notes
- benchmark contract proposals
- dataset/provenance records
- artifact review summaries
- manuscript or documentation contributions

High-impact claims, datasets, release decisions, and authorship decisions require more than activity volume.

## 5. Rights and Limitations by Level

Basic participation may allow:

- opening issues
- proposing docs or code
- commenting on public discussions
- submitting pull requests

Higher-trust review may allow:

- reviewing claim language
- reviewing dataset/provenance records
- reviewing reproducibility surfaces
- advising on release readiness

Maintainer rights may include:

- triage authority
- merge authority
- release coordination
- enforcement of claim and data policies

No level grants permission to publish restricted data, claim validation, approve a public preprint, or make legal/licensing conclusions unless that authority is explicitly assigned.

## 6. Review / Approval Requirements by Contribution Type

Documentation changes need reviewer or maintainer review when they affect public claims.

Code changes need technical review and may need reproducibility review.

Benchmark changes need benchmark contract review.

Dataset/provenance changes need source/license and public-safe artifact review.

Claim changes need claim registry and evidence ladder review.

Paper-support changes need paper alignment and source-to-claim review.

Release changes need release review process checks.

## 7. What Does Not Grant Maintainer Rights

Maintainer rights are not granted automatically by:

- opening issues
- submitting pull requests
- being acknowledged
- contributing code
- contributing data
- reviewing a manuscript
- participating in wet-lab collaboration
- being affiliated with a university, company, nonprofit, or partner organization
- being listed on a paper

Maintainer rights require explicit project decision and trust assignment.

## 8. What Does Not Automatically Grant Paper Authorship

Paper authorship is not automatic from:

- GitHub pull requests
- issue comments
- documentation edits
- code contributions
- benchmark suggestions
- bug reports
- reviewer feedback
- informal scientific advice
- use of Permea docs or code
- wet-lab interest or preliminary conversations

Authorship requires the authorship policy, paper-specific contribution record, accountability, final approval, and manual decision.

## 9. Escalation Path from Contributor to Trusted Reviewer / Maintainer

A contributor may become a trusted reviewer through repeated high-quality contributions that show:

- technical care
- policy awareness
- reproducibility discipline
- claim-boundary discipline
- dataset/public-safe artifact caution
- constructive review behavior

Maintainer candidacy additionally requires:

- sustained stewardship
- ability to enforce project boundaries
- alignment with open-source-first and benchmark-first principles
- explicit approval by current project governance

## 10. Claim / Data / Release Risk Labels

Contribution review should label risk where relevant:

- low claim risk
- medium claim risk
- high claim risk
- dataset/source review required
- row-level artifact risk
- release review required
- paper alignment required
- security/sensitive disclosure risk

High-risk contributions should not be merged or released until the relevant review is complete.

## Claim-Boundary Reminder

Permea may use "AlphaFold for Delivery" as ambition or positioning only.

Permea must not claim AlphaFold-level performance, adoption, or standardization; completed wet-lab validation unless documented; clinical efficacy; universal delivery prediction; production-grade drug delivery platform status; or dataset redistribution permission without source/license approval.

Benchmark claims must remain scoped to dataset, split, metric, model, artifact, and evidence level.

Dry-lab contributions do not equal biological validation. Wet-lab collaboration interest does not equal wet-lab validation.
