# Research Package Governance

Research package governance defines how Permea Core packages are proposed, assembled, reviewed, promoted, superseded, and archived.

## Proposal

A research package may be proposed when it has a clear purpose, package type, intended audience, expected evidence, expected dataset links, expected benchmark links, expected specification links, and claim boundary.

Proposed packages must remain clearly marked as proposed, not yet validated, and not a paper claim.

## Evidence Linking

Packages should link evidence records that support the package status. Evidence links should include limitations and should distinguish generated computational evidence from unsupported future validation.

## Dataset And Benchmark References

Packages should reference dataset and benchmark registry entries rather than implying independent dataset or benchmark status. Dataset and benchmark limitations carry into the research package.

## Claim Boundary Checks

Every package review should check:

- linked claim registry entries
- unsupported claim boundaries
- package lifecycle status
- whether public wording implies unsupported biological, clinical, or performance claims
- whether future-facing items are marked proposed, not yet demonstrated, out of scope, or requiring future validation

## Public Review Advancement

A package can advance toward public review only when evidence, dataset, benchmark, specification, reproducibility, validation, limitation, and claim-boundary links are complete for the stated scope.

## Superseded Packages

Superseded packages should retain the old package record, successor package, reason for replacement, affected evidence links, affected dataset links, affected benchmark links, and final validation state.

## Change Review

Research package changes should be reviewed when they alter package status, evidence links, benchmark links, dataset links, specification links, claims, validation path, reproducibility path, figures, tables, limitations, or public wording.

Required review checks:

- schema validity
- registry consistency
- assembly guide coverage
- linked evidence, benchmark, dataset, specification, and claim consistency
- claim-boundary scan
- public-safety scan
- local validation command output
